## Contexto

A tabela `produtos` é **foto-cêntrica**: 1 linha por foto. Um produto com 3 fotos ocupa 3 linhas. Isso causa:

1. **Copies infladas**: `generate-copies.ts` busca 896 linhas mas há ~426 SKUs únicos. Paga 2× e grava o último resultado (os anteriores são sobrescritos pelo UPSERT).
2. **Gabi API quebrada**: `GET /api/gabi/produto/[sku]` usa `.single()` — falha se o SKU tem >1 foto.
3. **Enriquecimento redundante**: `enrich-marketing.ts` reprocessa o mesmo SKU N vezes.
4. **Contagem errada**: relatórios mostram "896 produtos" quando há ~426 produtos reais.

---

## User Story

Como sistema, quero que `produtos` tenha **1 linha por SKU único** e que as fotos vivam numa tabela separada `produto_imagens`, para que copies, enriquecimento e Gabi API operem corretamente em nível de produto.

## Diagrama do Novo Modelo

```
produtos (master — 1 por SKU)
  ├── sku TEXT PRIMARY KEY
  ├── nome_produto TEXT
  ├── categoria TEXT
  ├── subcategoria TEXT
  ├── cor_dominante TEXT
  ├── material_aparente TEXT
  ├── tags TEXT[]
  ├── descricao_marketing TEXT   ← input para generate-copies.ts
  ├── descricao_tecnica TEXT
  ├── criado_em TIMESTAMPTZ
  └── atualizado_em TIMESTAMPTZ

produto_imagens (fotos — N por SKU)
  ├── id BIGSERIAL PRIMARY KEY
  ├── sku TEXT → FK produtos.sku
  ├── image_url TEXT
  ├── angulo TEXT
  ├── fundo TEXT
  ├── qualidade_foto TEXT
  ├── cor_dominante TEXT         ← override por foto
  ├── material_aparente TEXT     ← override por foto
  ├── problemas_foto TEXT[]
  ├── precisa_revisao BOOLEAN
  ├── hash_sha256 TEXT UNIQUE    ← dedup de upload
  ├── arquivo_original TEXT
  ├── processado_em TIMESTAMPTZ
  └── criado_em TIMESTAMPTZ
```

## Plano de Migration (002-produto-imagens.sql)

**Sequência segura — sem perda de dados:**

```
1. Criar produto_imagens (vazia)
2. Migrar todos os dados de produtos → produto_imagens
3. Criar produtos_master com DISTINCT ON (sku) de produtos
4. Adicionar FK produto_imagens.sku → produtos_master.sku
5. Renomear: produtos → produtos_legacy, produtos_master → produtos
6. Ativar RLS nas novas tabelas
```

A tabela `produtos_legacy` fica como backup. Apagar manualmente após validação em produção.
