## Contexto

O LBG-NEXT-INTEGRATION.md (§3.2) documenta que o Pix Agent precisa de dois campos para montar o `ProductData` completo:

- `metadados.linha` — ex: "Venezia", "Ipanema", "Carrara" (linha do produto)
- `metadados.aplicacao` — ex: ["piscina", "fachada", "parede"]

Esses campos **não existem como colunas no DB** e adicionar colunas novas exigiria enriquecimento manual de 426 SKUs. No entanto:

1. **`linha`** está embutida no `nome_produto` — ex: "Pastilha **Venezia** Marrom Cobre" → linha = "Venezia"
2. **`aplicacao`** pode ser derivada de `tags[]` — ex: `["piscina", "fachada"]` já existem como tags em muitos produtos

Solução: ensinar o Pix a extrair esses campos dos dados que já recebe, sem aguardar alteração de DB.

---

## User Story

Como Pix Agent (product-specialist), quero extrair automaticamente `linha` do `nome_produto` e derivar `aplicacao` dos `tags[]` do produto, para que o ProductData entregue ao Copi e ao Vix tenha `metadados.linha` e `metadados.aplicacao` preenchidos sem depender de campos novos no banco de dados.

---

## Acceptance Criteria

### Extração de linha (metadados.linha)

- [ ] AC1: `product-specialist.md` documenta padrão de extração de `linha` do `nome_produto`:
  - Formato típico: `"{Tipo} {Linha} {Cor/Modelo}"` — ex: "Pastilha Venezia Marrom Cobre"
  - Linhas conhecidas: Venezia, Ipanema, Carrara, Arabesco, Grigio, Branco, Metropolitan, Iracema (extrair por correspondência ou posição)
  - Fallback: `null` se nenhuma linha conhecida for identificada
- [ ] AC2: Campo `metadados.linha` no ProductData é populado com o valor extraído (string ou null)
- [ ] AC3: Extração é case-insensitive e trimada

### Derivação de aplicação (metadados.aplicacao)

- [ ] AC4: `product-specialist.md` documenta mapeamento de `tags[]` para `metadados.aplicacao`:
  - Tags de aplicação: `["piscina", "fachada", "parede", "piso", "banheiro", "cozinha", "área externa", "churrasqueira"]`
  - Pix filtra `tags[]` do produto mantendo apenas as tags de aplicação → array resultante = `aplicacao`
  - Fallback: `null` se nenhuma tag de aplicação estiver presente
- [ ] AC5: Campo `metadados.aplicacao` no ProductData é array de strings ou null
- [ ] AC6: Tags não-aplicação (`["cuba", "cerâmica", "vitrificada"]`) são ignoradas no filtro

### Contrato de saída

- [ ] AC7: ProductData normalizado pelo Pix inclui:
  ```yaml
  metadados:
    linha: "Venezia"          # string extraída ou null
    aplicacao: ["piscina"]    # array filtrado de tags ou null
  ```
- [ ] AC8: Campos existentes no ProductData não são alterados (sem breaking changes)
- [ ] AC9: Se API retornar produto sem `tags` (null/undefined) → `aplicacao: null` sem erro

### Documentação

- [ ] AC10: `squads/la-bella/LBG-NEXT-INTEGRATION.md` §3.2 atualizado: campo `metadados.linha` marcado como "extraído do nome_produto pelo Pix", `metadados.aplicacao` como "derivado de tags[] pelo Pix" — sem necessidade de coluna nova no DB

## Notas Técnicas

### Lógica de extração de linha (AC1) — implementar no Pix como instrução YAML

```yaml
extracao_linha:
  estrategia: correspondencia_lista_conhecida
  linhas_conhecidas:
    - Venezia
    - Ipanema
    - Carrara
    - Arabesco
    - Grigio
    - Metropolitan
    - Iracema
  instrucao: |
    Para extrair linha:
    1. Verificar se alguma das linhas_conhecidas aparece em nome_produto (case-insensitive)
    2. Se sim → metadados.linha = nome_da_linha (casing original da lista)
    3. Se não → metadados.linha = null
```

### Lógica de derivação de aplicação (AC4) — implementar no Pix como instrução YAML

```yaml
derivacao_aplicacao:
  tags_de_aplicacao:
    - piscina
    - fachada
    - parede
    - piso
    - banheiro
    - cozinha
    - área externa
    - area externa
    - churrasqueira
    - jardim
  instrucao: |
    Para derivar aplicacao:
    1. Receber tags[] do produto da API
    2. Filtrar: manter apenas tags que estão em tags_de_aplicacao (case-insensitive)
    3. Se array resultante não-vazio → metadados.aplicacao = array filtrado
    4. Se vazio ou tags=null → metadados.aplicacao = null
```

### ProductData — adição ao bloco normalizacao_api (product-specialist.md)

```yaml
normalizacao_api:
  mapeamento:
    sku             → product_id
    nome_produto    → nome
    image_url       → imagem_url
    cor_dominante   → cores_dominantes    # string → [string]
    material_aparente → acabamento
  campos_derivados:
    metadados.linha       → extraído de nome_produto via extracao_linha
    metadados.aplicacao   → derivado de tags[] via derivacao_aplicacao
  campos_sem_equivalente_na_api:
    dimensoes: null
```
